# BUGS_POSTMAN.md

**Дата:** 25 ноября 2025

## 1. Несоответствие имени параметра sellerID / sellerId (High)

**Описание бага**
POST-запрос `/api/1/item` возвращает идентификатор объявления внутри строки в поле `status`, вместо отдельного поля `id` в JSON.

**Шаги воспроизведения**

1. Сделать POST-запрос `/api/1/item` с корректным JSON, например:

```json
{
  "sellerID": 192348,
  "name": "Burger",
  "price": 300,
  "statistics": {
    "likes": 4,
    "viewCount": 200,
    "contacts": 2
  }
}
```

2. Посмотреть ответ сервера.

**Ожидаемое поведение**
Сервер возвращает JSON с отдельным полем `id`:

```json
{
  "status": "ok",
  "result": {
    "id": "<UUID>"
  }
}
```

**Фактическое поведение**
Сервер возвращает строку внутри поля `status`:

```json
{
  "status": "Сохранили объявление - ecdb317-884a-4683-90f3-f7b0d039df25"
}
```

**Приоритет:** High
**Обоснование:**
Невозможность корректного получения ID объявления нарушает работу клиента, автоматизацию тестов и интеграцию с другими сервисами.

## 2. Примеры ошибок содержат случайные lorem-ipsum ключи (Medium)

**Описание бага**  
В ошибках 400 используются неосмысленные ключи `culpa_b92`, `enim_24f`, `mollit_aa`.

**Шаги воспроизведения**

1. Открыть любой запрос, выбрать Response → Bad Request
2. Проверить поле `messages`

**Ожидаемое поведение**  
Понятные и стандартные ключи ошибок.

**Фактическое поведение**  
Случайные placeholder-ключи.

**Приоритет:** Medium  
**Обоснование:** Примеры не могут быть использованы в тестировании.

## 3. Ответ 100 Continue в GET statistic (Low)

**Описание бага**  
Присутствует пример ответа `100 Continue`, не являющийся валидным финальным ответом API.

**Шаги воспроизведения**

1. Найти GET statistic
2. Проверить блок Response

**Ожидаемое поведение**  
Ответы: 200, 400, 404, 500.

**Фактическое поведение**  
Содержится ответ 100 Continue.

**Приоритет:** Low  
**Обоснование:** Некритично, но путает разработчиков.
